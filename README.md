# Code Review: ProcessIncomingCallJob

---

### КРИТИЧЕСКИЕ

#### 1. Race condition при выборе оператора (нет блокировки)

```php
// Проблема: два воркера одновременно видят одного оператора как доступного
$operator = Operator::where('available', true)
    ->orderBy('last_call_at')
    ->first();

$operator->available = false;
$operator->save(); // оба воркера сохраняют одного и того же оператора
```

**Что происходит:** при параллельной обработке двух звонков оба воркера выбирают одного оператора (он ещё доступен в момент SELECT обоих), оба ставят `available = false`, оба назначают звонки. Один оператор получает два звонка.

**Исправление:** SELECT ... FOR UPDATE внутри транзакции:
```php
DB::transaction(function () use ($call) {
    $operator = Operator::where('available', true)
        ->orderBy('last_call_at')
        ->lockForUpdate()
        ->first();
    // ...
});
```

#### 2. Race condition на самом звонке (нет блокировки записи Call)

```php
$call = Call::find($this->callId);

if ($call->status === 'new') { // проверка без блокировки
    // ...
}
```

**Что происходит:** если одна и та же задача попала в очередь дважды (дублирование в Redis, баг при dispatch), два воркера читают `status = 'new'`, оба проходят проверку, оба назначают разных операторов на один звонок.

**Исправление:** читать запись с `lockForUpdate()` внутри транзакции:
```php
$call = Call::where('id', $this->callId)
    ->lockForUpdate()
    ->first();
```

#### 3. Нарушение идемпотентности при повторах (дублирование вызова телефонии)

```php
// Сценарий: DB-сохранение прошло, HTTP-запрос падает с исключением
// Job повторяется (tries = 5), status уже 'assigned' => проверка не сработает
// Но если исключение прилетело до сохранения, status ещё 'new' => телефония вызовется снова

app(TelephonyClient::class)->sendCallAssigned($call->id, $operator->id);
```

**Что происходит:** в зависимости от того, на каком шаге упало, статус может быть `new` или `assigned`, оператор может быть уже занят или нет, а телефония уже вызвана или нет. 
Нет единой точки консистентности.

**Исправление:**
- Всё изменение состояния (оператор + звонок) в одной транзакции.
- Вызов телефонии вне транзакции, после коммита.
- Идемпотентность телефонии отдельный статус (`telephony_sent`) или запись в отдельной таблице.

#### 4. Нет DB-транзакции (частичное сохранение оставляет систему в несогласованном состоянии)

```php
$operator->available = false;
$operator->save(); // сохранили

$call->operator_id = $operator->id;
$call->status = 'assigned';
$call->save(); // если вылетит здесь оператор занят, но звонок не назначен
```

**Что происходит:** оператор отмечен как занятый, но звонок так и остался `new`. При повторе задача попытается найти нового доступного оператора (первого уже нет), а занятый оператор висит в состоянии `available = false` навсегда.

**Исправление:** транзакция вокруг обоих `save()`.

### ВАЖНЫЕ

#### 5. `last_call_at` оператора не обновляется (балансировка не работает)

```php
$operator->available = false;
$operator->save(); // last_call_at не трогается
```

`orderBy('last_call_at')` призван обеспечить round-robin по операторам, но если поле не обновляется — всегда выбирается один и тот же оператор (с самым старым временем).

**Исправление:**
```php
$operator->available = false;
$operator->last_call_at = now();
$operator->save();
```

#### 6. Повтор на "нет операторов" (бессмысленный немедленный retry)

```php
public $tries = 5;

// ...
throw new \Exception('No available operators');
```

Если операторов нет, немедленный повтор через секунду тоже не найдёт операторов. Пять попыток сжигаются впустую, после чего задача уходит в failed_jobs.

**Исправление:** использовать `release()` с задержкой и ограничить повторы для этого конкретного случая:
```php
if (!$operator) {
    $this->release(30); // вернуть в очередь через 30 секунд
    
    return;
}
```
Или выделить кастомное исключение и настроить `retryUntil()`.

#### 7. Выброс базового `\Exception` вместо типизированных исключений

```php
throw new \Exception('No available operators');
```

Нельзя различить ожидаемую бизнес-ситуацию (нет операторов) от программной ошибки или сетевого сбоя. Все идут в одну корзину повторов.

**Исправление:**
```php
class NoAvailableOperatorException extends \RuntimeException {}
class TelephonyException extends \RuntimeException {}
```

В Job:
```php
public function retryUntil(): \DateTime
{
    return now()->addMinutes(10);
}
```

#### 8. Нет логирования промежуточных состояний и ошибок

Логируется только успешное назначение. Нет записи: клиент не найден, оператор не найден, ошибка телефонии, повтор задачи. В production под нагрузкой отлаживать инциденты будет крайне сложно.

**Исправление:**
```php
Log::warning('Client not found for call', ['phone' => $call->phone, 'call_id' => $call->id]);
Log::warning('No available operators, releasing job', ['call_id' => $call->id]);
Log::error('Telephony request failed', ['call_id' => $call->id, 'error' => $e->getMessage()]);
```

### БЫЛО БЫ ХОРОШО СДЕЛАТЬ

#### 9. Нет строгой типизации и type hints

```php
private $callId; // нет типа

public function __construct($callId) // нет типа параметра
public function handle() // нет возвращаемого типа
```

**Исправление:**
```php
public function __construct(private readonly int $callId) {}
public function handle(): void {}
```

#### 10. Нет exponential backoff для повторов

`$tries = 5` без задержки означает немедленные повторы. При сетевом сбое телефонии пять запросов прилетят подряд за секунду.

**Исправление:**
```php
public function backoff(): array
{
    return [10, 30, 60, 120]; // секунды задержки между попытками
}
```

## Тесты

Приоритет по риску для production:

### 1. Идемпотентность (критично)
```
- Тест: запустить Job дважды с одним callId > звонок назначен ровно одному оператору
- Тест: Job с уже назначенным звонком (status != 'new') > ничего не меняется
```

### 2. Параллельная обработка / race condition (критично)
```
- Тест: два Job запущены одновременно с разными callId > каждый получает разного оператора
- Тест: один оператор доступен, два звонка одновременно > один назначен, второй retried
```

### 3. Retry-логика при отсутствии операторов (важно)
```
- Тест: нет доступных операторов > Job released с задержкой, не failed
- Тест: оператор освободился между попытками > назначается на повторе
```

### 4. Ошибка телефонии (важно)
```
- Тест: TelephonyClient бросает исключение > DB-состояние не изменено или задача корректно retried
- Тест: телефония недоступна > оператор не остаётся навечно занятым
```

### 5. Поиск клиента (норм, но важно для бизнеса)
```
- Тест: клиент найден > call.client_id заполнен
- Тест: клиент не найден > звонок всё равно назначается, client_id = null, есть warning-лог
```

## Что я бы НЕ стал делать прямо сейчас

1. **Полный рефакторинг в event-sourcing или CQRS** система уже в production, риск регрессий высок, а профит неочевиден без более глубокого понимания всей архитектуры.

2. **Выносить логику выбора оператора в отдельный микросервис** преждевременная оптимизация. Сначала зафиксировать текущие баги, потом думать о масштабировании.

3. **Менять схему очереди с Redis на другой брокер** не решает найденные проблемы, но добавляет новую поверхность для ошибок.

4. **Добавлять Saga/Outbox pattern сразу** это правильное долгосрочное решение для консистентности с телефонией, но требует изменений схемы БД и инфраструктуры. Сначала нужно зафиксировать критические баги минимальными изменениями.

5. **Переписывать логику клиента** `Client::where('phone', ...)` выглядит корректно, телефон как идентификатор приемлемо для этого контекста.

## Допущения и риски из-за неопределённости

### Допущения

| Область | Допущение |
|---|---|
| Телефония | `sendCallAssigned` синхронный HTTP-запрос. Нет гарантий идемпотентности на стороне телефонии |
| Redis | Возможно дублирование задач (at-least-once delivery) |
| Операторы | `available = false` выставляется только этим Job; освобождается другим механизмом (вебхук от телефонии?) |
| `last_call_at` | Обновляется где-то ещё или вообще не используется в текущем коде |
| `Call.phone` | Всегда нормализован (без пробелов, скобок, +7/8) |

### Риски

1. **Оператор навсегда "занят"** если Job упал после `operator.save()` и до `call.save()`, оператор останется `available = false`. Нет механизма автоматического освобождения.

2. **Дублирование в телефонии** если телефония получила запрос, но вернула 5xx, Job повторится и отправит запрос снова. Если телефония не идемпотентна возможен двойной звонок оператору.

3. **Утечка операторов при масштабировании** чем больше воркеров, тем выше вероятность race condition. Без транзакции с блокировкой проблема растёт линейно с числом воркеров.

4. **Тихая потеря звонка** если все 5 попыток исчерпаны (напр. телефония стабильно недоступна), звонок уходит в `failed_jobs` и о нём никто не узнает без алертинга на этот канал.

5. **`call.phone` как внешний ключ без нормализации** если телефон пришёл в формате `+7(999)` а в базе `79999...`, клиент не найдётся. Нет явной нормализации ни при сохранении звонка, ни при поиске.

## Дополнительно

### 1. `LoggerInterface` убран из конструктора Job перенесён в `handle()`

**Причина:** Monolog-хендлеры (файловые, DB) содержат несериализуемые ресурсы. При диспатче Job сериализуется в Redis всё в конструкторе попадает в очередь. Это либо вызывает исключение при сериализации, либо логгер восстанавливается в некорректном состоянии.

**Решение:** `handle()` принимает `LoggerInterface $logger` Laravel IoC резолвит его из контейнера на воркере в момент выполнения.

**Нюанс:** `failed()` вызывается Laravel вне DI-цикла. Инъекция параметров там невозможна по дизайну фреймворка используется `logger()`-хелпер. Это задокументировано в коде.

### 2. `CallStatus` enum вместо магических строк

**Добавлен:** `app/Enums/CallStatus.php` (backed enum: `New = 'new'`, `Assigned = 'assigned'`, `Failed = 'failed'`).

Используется через `$casts` в модели `Call` Eloquent автоматически конвертирует при чтении/записи. Сравнение `$call->status !== CallStatus::New` типобезопасно и не допускает опечаток.

### 3. Методы `Operator::markBusy()` и `Call::assignTo()`  инкапсуляция состояния

**Причина:** `OperatorAssignmentService` знал о внутренней структуре моделей — нарушение инкапсуляции ("Tell, Don't Ask").

**Решение:**
- `Operator::markBusy()` устанавливает `available = false`, обновляет `last_call_at`, сохраняет
- `Call::assignTo(Operator $operator)` назначает оператора, меняет статус на `Assigned`, сохраняет

Сервис теперь читается как намерение, а не как императивный набор присваиваний:
```php
// До
$operator->available    = false;
$operator->last_call_at = now();
$operator->save();
$call->operator_id = $operator->id;
$call->status      = 'assigned';
$call->save();

// После
$operator->markBusy();
$call->assignTo($operator);
```
